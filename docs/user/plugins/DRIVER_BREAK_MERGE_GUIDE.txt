What this series is
-------------------

The Driver Break plugin started as one large tree under navit/plugin/driver_break/.
I split it into six reviewable units that still become a single Navit OSD plugin
(libosd_driver_break.so) once they are all on trunk in the right order.

Each PR branch carries three commits: the plugin slice and tests, the docs for
that unit, and the shared Qt5/po/workflow formatting that upstream CI expects.
The commit list is intentionally short; review the diff against trunk rather than
counting commits.

Core holds the split driver_break_db_*.c sources, fuel_core, osd_lifecycle,
route/finder/hiking/cycling/energy modules, and plugin_init().

OSD holds driver_break_osd_{core,car,truck,motorcycle,hiking,cycling,fuel,
overnight}.c plus driver_break_gui_shim.c.

Core must merge after OSD: lifecycle in core calls driver_break_register_commands(),
which is defined in driver_break_osd_core.c on the OSD branch.


Branch isolation vs trunk
-------------------------

Each PR branch is built to compile its slice alone on CI:

  - Stubs stand in for symbols from other units (for example
    driver_break_slice_stubs.c on core).
  - USE_PLUGINS=n skips plugin slices on Sailfish harbour builds.
  - feature-core also requires SQLITE3_FOUND.

Those stubs are scaffolding for review builds, not the long-term layout. After
the sequential merge, drop the stubs and link one plugin_driver_break target from
all units. When the first PR lands the shared Qt5/po formatting, later PRs can
drop duplicate formatting hunks during conflict resolution.

  cmake --build build --target <slice_test_target> -j$(nproc)
  ctest -R 'ecu_|driver_break_geo|driver_break_srtm|driver_break_poi|
           driver_break_config|driver_break_db' --output-on-failure
